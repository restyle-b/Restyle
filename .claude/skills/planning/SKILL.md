# Skill: תכנון (Planning)

מטרת הסקיל: להבטיח שכל פיתוח מתחיל בתכנון מסודר, ומסתיים בעדכון הרודמאפ.
אתה האדריכל. אל תקפוץ לקוד לפני שהתכנון ברור.

## מתי להפעיל
- לפני התחלת אפיק/פיצ'ר חדש (חנות, אקדמיה, הזמנות, אזור אישי, אדמין, תשלומים).
- כשהמשתמש מבקש "לתכנן" או שואל "איך נבנה את X".
- בתחילת שלב (Phase) חדש ברודמאפ.

## תהליך התכנון (חובה לעבור על כל הסעיפים)

1. **הבנת הדרישה** — מה הערך למשתמש הסופי (לקוחה / תלמיד / מנהל)? מי הפרסונה?
2. **הקשר במערכת** — איך זה מתחבר לקיים? עיין ב-`docs/ARCHITECTURE.md` לפני כל החלטה.
3. **מודל נתונים** — אילו טבלאות/שדות/יחסים ב-Prisma? שרטט את הסכמה.
4. **חוזה API / Server Actions** — אילו endpoints/actions? קלט, פלט, ולידציה (zod).
5. **UI/UX** — אילו מסכים/קומפוננטות? RTL + עברית. השראה מ-menspire.com (פרימיום, נקי, מינימליסטי, תמונות גדולות).
6. **אבטחה** — מי מורשה? איזה מידע רגיש? הפעל את סקיל `security` לבדיקה.
7. **קריטריוני קבלה (Acceptance Criteria)** — רשימת בדיקות שצריכות לעבור. אלה הופכים למשימות QA.
8. **פירוק למשימות** — רשימת steps קטנים וניתנים לבדיקה.
9. **עדכון רודמאפ** — הוסף/עדכן את `ROADMAP.md` עם הפיצ'ר והמשימות.

## עקרונות תכנון לפרויקט
- **Mobile-first ו-RTL-first** — רוב הלקוחות יגיעו מהנייד, הכל בעברית.
- **שכבת הפשטה לתשלומים** — אף פעם לא לקודד ספק תשלום ישירות. תמיד דרך `PaymentProvider` interface (ראה `docs/ARCHITECTURE.md`).
- **הפרדת תחומים** — marketing / shop / academy / account / admin הם מודולים נפרדים.
- **Server-first** — העדף React Server Components ו-Server Actions; client רק כשצריך אינטראקטיביות.
- **כל החלטה ארכיטקטונית מתועדת** ב-`docs/ARCHITECTURE.md` עם נימוק.

## פלט הסקיל
מסמך תכנון קצר (יכול להיות בצ'אט או ב-`docs/features/<name>.md`) הכולל: מודל נתונים, חוזה API, מסכים, קריטריוני קבלה, ורשימת משימות — ועדכון מקביל ב-`ROADMAP.md`.

## חיבור לסקילים אחרים
- בסיום תכנון → מעבר ל-`development`.
- בכל תכנון הכולל נתוני משתמש/תשלום → התייעצות עם `security`.
- קריטריוני הקבלה שהוגדרו כאן → הקלט של `qa`.
