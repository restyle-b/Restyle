# Skill: בדיקות ואיכות (QA)

מטרה: לוודא שכל פיצ'ר עומד בקריטריוני הקבלה לפני שמסמנים "בוצע" ברודמאפ.
לא מסמנים משימה כהושלמה בלי שעברה QA.

## מתי להפעיל
- אחרי כל פיתוח פיצ'ר, לפני commit/סיום.
- כשהמשתמש שואל "האם זה עובד" / "תבדוק".
- לפני סיום שלב (Phase) ברודמאפ.

## רמות בדיקה
1. **Static** — `npm run lint` + `npm run typecheck` נקיים.
2. **Unit (Vitest)** — לוגיקה עסקית: חישובי מחיר/עגלה, ולידציות zod, מתאמי תשלום (mock), helpers.
3. **Integration** — server actions מול DB בדיקה (חישוב הזמנה, שינוי סטטוס, הרשאות).
4. **E2E (Playwright)** — מסעות משתמש מלאים: הרשמה→התחברות→קנייה, הרשמה לקורס, ניהול הזמנה באדמין.
5. **Manual / Exploratory** — לפי checklist למטה.

## Checklist ידני לכל פיצ'ר
- [ ] **קריטריוני קבלה** — כל אחד מהם מתקיים בפועל.
- [ ] **RTL** — פריסה תקינה מימין לשמאל, אין טקסט/אייקונים הפוכים.
- [ ] **רספונסיביות** — מובייל (360px), טאבלט, דסקטופ.
- [ ] **מצבי קצה** — רשימה ריקה, שגיאת רשת, קלט לא תקין, כפול-לחיצה, מלאי אזל.
- [ ] **הרשאות** — משתמש לא מחובר / משתמש אחר / אדמין — כל אחד רואה רק את המותר.
- [ ] **נגישות** — ניווט מקלדת, alt לתמונות, ניגודיות, ARIA בסיסי.
- [ ] **ביצועים** — אין תמונות ענק, אין N+1 queries, Lighthouse סביר.
- [ ] **שגיאות למשתמש** — הודעות ברורות בעברית, ללא קריסות לבנות.

## מוקדי בדיקה קריטיים (לפי דומיין)
- **חנות/צ'קאאוט:** חישוב סכום נכון (אגורות), מלאי, קופונים, חזרה כושלת מתשלום, idempotency של callback.
- **הזמנות:** מעברי סטטוס תקינים, היסטוריה למשתמש נכון, אדמין מעדכן בלי לדרוס.
- **Auth/אזור אישי:** אי-אפשר לגשת לנתוני משתמש אחר; session, logout, reset.
- **תשלומים:** מתאם נבדק מול mock; webhook מאומת חתימה; כפילויות נחסמות.

## פלט
דוח קצר: מה נבדק, מה עבר, מה נכשל, ובאגים פתוחים. רק אם הכל ירוק → לסמן ✅ ב-`ROADMAP.md`.
אם נמצא באג → לתעד ולחזור ל-`development` (לא לסמן הושלם).
